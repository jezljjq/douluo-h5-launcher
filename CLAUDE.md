## 一、项目定位

- 当前版本：**前台串行稳定版（当前窗口尺寸）+ 窗口管理区**
- 当前支持：方式一通行证上号、方式二账号密码 + 通行证上号、单账号运行、当前层串行、全部串行
- 当前窗口管理区支持：批量启动、识别、排列、关闭、标题模板、重命名、排列后自动编号标题、参数记忆
- 当前暂不支持：真后台点击、真并发、遮挡点击、任意分辨率自适应
- 当前窗口尺寸已验证稳定，但后续游戏窗口尺寸可能变化——窗口尺寸自适应需要作为独立阶段开发，不要在当前 bug 修复中顺手改

## 二、核心对象关系

收藏编号 ≠ 游戏窗口号。

| 层级 | 收藏编号 | 游戏窗口号 |
|------|---------|-----------|
| 第一层 | 1-8 | 1-8 |
| 第二层 | 1-8 | 9-16 |
| 第三层 | 1-8 | 17-24 |
| 第四层 | 1-8 | 25-32 |

规则：
- 收藏编号只用于读取收藏夹链接
- 游戏窗口号用于定位登录程序窗口
- OCR、点击、状态判断都基于游戏窗口号
- 第二层收藏1对应游戏窗口9，不是窗口1

## 三、窗口定位规则

当前窗口标题类似 `斗罗大陆H5-9-伊号科技`，通过标题中的 `H5-{游戏窗口号}` 匹配。

但后续窗口标题可能变化（可能所有窗口只显示 `斗罗大陆H5`），所以不能写死完整标题。

**当前方案（已验证）**：标题含 `H5-{编号}-` 匹配

**窗口管理模块当前状态**：窗口管理功能已经从独立工具迁入上号器内部模块。上号器不应依赖独立项目 `DLH5WindowManager` 的路径。独立项目被移动或删除，理论上不应影响上号器内部窗口管理功能。

**后续兼容方案（设计已记录，当前不改）**：
1. 枚举所有登录程序窗口
2. 过滤标题含"斗罗大陆H5"
3. 过滤类名 `WindowsForms10.Window.8.app.*`
4. 过滤尺寸接近 320×540
5. 按坐标排序：先按 top 分行，每行按 left 从左到右
6. 映射：第一行→窗口1-8，第二行→窗口9-16，依此类推

## 四、两种上号方式

### 方式一：通行证上号（当前主流程 ✅）

**状态**：已开发，正在持续稳定和优化。当前优先级最高。

流程：
1. 定位登录程序窗口
2. 图像特征判断页面状态（`detect_login_page_state`）
3. OCR/图像识别"本次通行证"，提取8位hex
4. 打开浏览器游戏页
5. 关闭公告
6. 模板匹配 + Dm点击"通行证"按钮
7. Dm输入通行证（clip.exe + Ctrl+V）
8. 视觉定位 + Dm点击确认按钮
9. 校验登录程序窗口QR码是否消失

需要持续保证：
- OCR通行证识别稳定
- 已登录窗口正确跳过
- 二维码页不能误判已登录
- 公告关闭稳定
- 通行证按钮定位稳定
- 输入/确认稳定
- 串行（单账号/当前层/全部）稳定
- 日志、截图、打包流程稳定

### 方式二：账号密码 + 通行证上号（已开发 ✅）

**状态**：已开发，支持 CSV 导入、账号密码登录、通行证输入和串行批量。

CSV 表头：`name,url,username,password`

当前流程：
1. 从登录程序窗口获取通行证（复制优先，OCR 兜底）
2. 打开 CSV 中的 url
3. 输入 username/password 登录
4. 关闭公告
5. Dm 点击通行证按钮
6. 输入通行证并确认
7. 校验登录程序窗口 QR 码是否消失

**注意**：password禁止打印到日志，GUI只显示"已填写/未填写"。

## 五、OCR正式方案

**当前正式方案**：
- 登录程序窗口截图 → 判断二维码页 → 优先双击选中 8 位通行证并 `Ctrl+C` 复制 → 读取剪贴板
- 复制失败后才进入 OCR 兜底
- OCR 低置信度不得继续输入通行证
- OCR 出现 `c/e` 等混淆时必须失败或进入人工处理，不能假成功

OCR 风险示例：

- 真实：`4425cbaa`
- 错误：`4425ebaa`
- 错误位置：`c` 被识别为 `e`

OCR 兜底必须输出候选结果、候选票数、字符级投票、存疑位置、是否接受、失败类型。新增失败类型：`OCR_LOW_CONFIDENCE`、`OCR_AMBIGUOUS_CHAR`。

原则：宁可真实失败，也不要错误输入通行证。

**禁止回退到**：
- 二维码定位裁剪OCR
- 浏览器页面OCR
- URL解析通行证
- iframe/DOM读取通行证

**QR解码说明**：
- OpenCV QRCodeDetector已验证暂不可用（解码成功率低）
- 图像特征（black_ratio + edge_density + local_variance）可以用于判断是否为二维码登录页
- 但不要依赖QR解码获取通行证内容

## 六、登录程序窗口状态判断（关键）

必须区分三种状态：`qr_page` / `logged_in` / `unknown`。

**禁止逻辑**：`OCR失败 = 已登录`

**正确逻辑**：

| 状态 | 行为 |
|------|------|
| `logged_in` | 标记"已登录，跳过"，不跑OCR |
| `qr_page` | 必须识别通行证；OCR失败→标记"二维码页通行证识别失败"，不判已登录 |
| `unknown` | 保守策略，失败保留现场，不直接判成功 |

**注意**：窗口1-8可能是已登录界面，窗口9以后可能是二维码登录界面。不能把窗口9以后错误标记为"已登录，跳过"。

## 七、点击方案

**当前正式方案**：Dm无绑定前台点击（MoveTo + LeftDown/LeftUp）

**当前限制**：移动真实鼠标、不支持后台、不支持并发、不支持遮挡

**已验证不可用，禁止重复尝试**：
- Playwright canvas click（isTrusted拦截）
- CDP dispatchMouseEvent（isTrusted拦截）
- SendMessage/PostMessage（Chrome GPU渲染绕过）
- Dm BindWindow（7.2607 + Win11全模式崩溃）

除非环境明显变化（大漠新版本、Chrome架构变化），否则不要反复尝试。

### 停止任务与关闭程序

停止任务机制已修复并验证通过：

- 点击“停止任务”后，不再只是设置 `stop_event`。
- 会强制终止当前账号运行子进程。
- 会清理 `dm_click_helper.py` 子进程。
- 会清理本次 Playwright/Chromium 相关进程。
- 不再继续后续账号。

关闭程序机制已修复并验证通过：

- 点击窗口右上角关闭时，会先执行停止任务和子进程清理。
- GUI 关闭后，不允许残留子进程继续移动鼠标。
- 避免 `_drain_ui_queue` 在窗口关闭后继续 `after` 回调。

禁止事项：

- 禁止只做协作式 `stop_event`，不清理实际子进程。
- 禁止误杀所有 `python.exe`，只清理本项目相关子进程。

## 十三、`subprocess.Popen` monkey-patch 规则

`automation.py` 模块级 monkey-patch 覆盖 `subprocess.Popen` 用于注入 `CREATE_NO_WINDOW`。

**必须遵守**：
- 必须用 class 继承（`class _NoConsolePopen(_original_popen)`），禁止用 function 替换
- `from playwright.sync_api import sync_playwright` 前必须临时恢复原始 Popen，导入后恢复补丁
- 违反会导致 `TypeError: function() argument 'code' must be code, not str`
