# 斗罗大陆H5上号器 — 项目级规则

> 全局通用开发纪律见 `C:\Users\Administrator\.claude\CLAUDE.md`。
> 本文档只记录本项目特有的业务规则、当前状态、稳定方案、禁止回退方案。

---

## 一、项目定位

- 当前版本：**前台串行稳定版（当前窗口尺寸）**
- 当前支持：单账号运行、当前层串行、全部串行
- 当前暂不支持：真后台点击、真并发、遮挡点击、任意分辨率自适应
- 当前窗口尺寸已验证稳定，但后续游戏窗口尺寸可能变化——窗口尺寸自适应需要作为独立阶段开发，不要在当前 bug 修复中顺手改

---

## 二、核心对象关系

收藏编号 ≠ 游戏窗口号。

| 层级 | 收藏编号 | 游戏窗口号 |
|------|---------|-----------|
| 第一层 | 1-8 | 1-8 |
| 第二层 | 1-8 | 9-16 |
| 第三层 | 1-8 | 17-24 |
| 第四层 | 1-8 | 25-32 |

规则：
- 收藏编号只用于读取收藏夹链接
- 游戏窗口号用于定位登录程序窗口
- OCR、点击、状态判断都基于游戏窗口号
- 第二层收藏1对应游戏窗口9，不是窗口1

## 三、窗口定位规则

当前窗口标题类似 `斗罗大陆H5-9-伊号科技`，通过标题中的 `H5-{游戏窗口号}` 匹配。

但后续窗口标题可能变化（可能所有窗口只显示 `斗罗大陆H5`），所以不能写死完整标题。

**当前方案（已验证）**：标题含 `H5-{编号}-` 匹配

**后续兼容方案（设计已记录，当前不改）**：
1. 枚举所有登录程序窗口
2. 过滤标题含"斗罗大陆H5"
3. 过滤类名 `WindowsForms10.Window.8.app.*`
4. 过滤尺寸接近 320×540
5. 按坐标排序：先按 top 分行，每行按 left 从左到右
6. 映射：第一行→窗口1-8，第二行→窗口9-16，依此类推

## 四、两种上号方式

### 方式一：通行证上号（当前主流程 ✅）

**状态**：已开发，正在持续稳定和优化。当前优先级最高。

流程：
1. 定位登录程序窗口
2. 图像特征判断页面状态（`detect_login_page_state`）
3. OCR/图像识别"本次通行证"，提取8位hex
4. 打开浏览器游戏页
5. 关闭公告
6. 模板匹配 + Dm点击"通行证"按钮
7. Dm输入通行证（clip.exe + Ctrl+V）
8. 视觉定位 + Dm点击确认按钮
9. 校验登录程序窗口QR码是否消失

需要持续保证：
- OCR通行证识别稳定
- 已登录窗口正确跳过
- 二维码页不能误判已登录
- 公告关闭稳定
- 通行证按钮定位稳定
- 输入/确认稳定
- 串行（单账号/当前层/全部）稳定
- 日志、截图、打包流程稳定

### 方式二：账号密码 + 通行证上号（暂未开发 ❌）

**状态**：暂未开发，属于后续计划。

开发前提：必须等方式一十分稳定后，再开始设计和开发。方式二不能插队。

计划中的CSV表头：`name,url,username,password`

计划流程（后续参考，当前不实现）：
1. OCR识别"本次通行证"（必须先OCR，不是只用账号密码）
2. 打开CSV中的url
3. 输入username/password登录
4. 后续复用方式一步骤5-9

**注意**：password禁止打印到日志，GUI只显示"已填写/未填写"。

**当前阶段不要**：
- 不要开发方式二
- 不要新增CSV登录逻辑
- 不要改账号密码登录流程

## 五、OCR正式方案

**当前正式方案**：
- 登录程序窗口截图 → OCR识别通行证区域 → 查找"本次通行证" → 提取8位hex
- 二维码页确认后，优先用底部文字区域OCR（避开QR码干扰），低置信度回退全图OCR
- 票数未过半的结果不信任，回退或标记失败

**禁止回退到**：
- 二维码定位裁剪OCR
- 浏览器页面OCR
- URL解析通行证
- iframe/DOM读取通行证

**QR解码说明**：
- OpenCV QRCodeDetector已验证暂不可用（解码成功率低）
- 图像特征（black_ratio + edge_density + local_variance）可以用于判断是否为二维码登录页
- 但不要依赖QR解码获取通行证内容

## 六、登录程序窗口状态判断（关键）

必须区分三种状态：`qr_page` / `logged_in` / `unknown`。

**禁止逻辑**：`OCR失败 = 已登录`

**正确逻辑**：

| 状态 | 行为 |
|------|------|
| `logged_in` | 标记"已登录，跳过"，不跑OCR |
| `qr_page` | 必须识别通行证；OCR失败→标记"二维码页通行证识别失败"，不判已登录 |
| `unknown` | 保守策略，失败保留现场，不直接判成功 |

**注意**：窗口1-8可能是已登录界面，窗口9以后可能是二维码登录界面。不能把窗口9以后错误标记为"已登录，跳过"。

---

## 七、点击方案

**当前正式方案**：Dm无绑定前台点击（MoveTo + LeftDown/LeftUp）

**当前限制**：移动真实鼠标、不支持后台、不支持并发、不支持遮挡

**已验证不可用，禁止重复尝试**：
- Playwright canvas click（isTrusted拦截）
- CDP dispatchMouseEvent（isTrusted拦截）
- SendMessage/PostMessage（Chrome GPU渲染绕过）
- Dm BindWindow（7.2607 + Win11全模式崩溃）

除非环境明显变化（大漠新版本、Chrome架构变化），否则不要反复尝试。

---

## 八、运行模式

- 当前只支持前台严格串行
- 禁止开启真并发（前台点击会抢鼠标、串窗口）
- 后续要做真并发，必须先解决后台点击或多实例隔离

## 九、窗口尺寸和分辨率

- 登录程序窗口约 320×540
- 浏览器viewport约 960×720
- 通行证按钮模板、公告关闭坐标、输入框/确认按钮定位均在当前尺寸下验证稳定

## 十二、后续开发优先级

1. **方式一稳定性修复**（当前在修）
   重点：登录程序窗口状态判断，避免二维码页误判已登录。

   包括单账号、当前层串行、全部串行。

   如果后续窗口标题不带编号，再单独开发窗口映射方案。

7. **窗口尺寸自适应**
   后续单独阶段处理。

   只有方式一十分稳定后再开发。当前不要开发。

   最后再研究，不影响当前前台串行稳定版。

## 十三、`subprocess.Popen` monkey-patch 规则

`automation.py` 模块级 monkey-patch 覆盖 `subprocess.Popen` 用于注入 `CREATE_NO_WINDOW`。

**必须遵守**：
- 必须用 class 继承（`class _NoConsolePopen(_original_popen)`），禁止用 function 替换
- `from playwright.sync_api import sync_playwright` 前必须临时恢复原始 Popen，导入后恢复补丁
- 违反会导致 `TypeError: function() argument 'code' must be code, not str`
