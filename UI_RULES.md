# UI 改造规则

## 1. 项目定位

本项目是斗罗大陆 H5 上号器 / 自动化控制台，不是娱乐网站，不做游戏风皮肤。

界面应服务于批量账号操作、窗口管理、状态追踪和异常处理，而不是强调娱乐化、装饰化或游戏化视觉。

## 2. UI 总目标

UI 改造目标是：

- 专业
- 紧凑
- 清晰
- 状态可读
- 异常明确
- 适合批量账号操作

所有界面调整都应优先提升操作效率、状态判断效率和批量任务可控性。

## 3. 推荐布局

推荐使用控制台式布局：

- 左侧：任务导航 / 模式选择 / 批次选择
- 中间：账号状态表格
- 右侧：当前账号流程详情
- 底部：日志摘要 / 错误摘要

如果阶段性仍使用上下布局，也必须保持信息层级清晰，避免控件松散堆叠。

## 4. 输入框规则

- 普通输入框高度控制在 28px - 34px。
- 路径输入框高度不得超过 36px。
- URL、账号、密码、通行证、路径输入框高度必须统一。
- 禁止出现过高、过宽、视觉比例失衡的输入框。
- 短字段必须使用固定宽度，例如窗口编号、数量、间隔、坐标、重试次数。
- 只有路径、URL、日志路径等长文本输入框允许横向扩展。

## 5. 按钮规则

- 主按钮、次按钮、危险按钮要有明确区分。
- 同一行按钮高度必须一致。
- 主要操作按钮包括：开始、暂停、停止、重试、打开日志。
- 危险操作必须弱化或二次确认。
- 停止、关闭、清理类按钮应使用红色或危险样式。
- 普通辅助按钮应避免比主要操作按钮更抢眼。

## 6. 状态颜色

- 成功：绿色
- 失败：红色
- 运行中：蓝色
- 等待：灰色
- 警告：橙色
- OCR/通行证异常：红色或橙色

状态颜色必须稳定统一，不能在不同区域表达相反含义。

## 7. 表格规则

账号状态表必须包含：

- 窗口编号
- 账号名称
- 当前阶段
- OCR 状态
- 通行证状态
- 登录结果
- 耗时
- 错误原因

表格应是主视觉区域之一。账号状态、失败原因和当前阶段必须一眼可见。

密码不得明文显示，只能显示“已填写 / 未填写”等脱敏状态。

## 8. 日志规则

- 主界面只显示摘要日志。
- 详细日志写入文件。
- 不允许主界面持续刷屏导致卡顿。
- 多窗口日志必须带窗口编号。
- 错误日志必须能区分 OCR、通行证、DM 点击、公告关闭、Playwright、登录校验、窗口管理等来源。

## 9. 禁止事项

UI 改造任务中禁止修改：

- OCR 识别
- 通行证提取
- DM 点击
- 公告关闭
- Playwright 初始化
- 登录流程
- 批量流程
- 打包脚本
- 配置读取逻辑

UI 改造不得顺手重构稳定业务模块，不得改变登录判断规则，不得改变批量任务执行语义。

## 9.1 允许修改范围

UI 改造任务原则上只允许修改界面层文件，例如：

- GUI 主窗口文件
- UI 布局文件
- 样式文件
- 前端组件文件
- 页面展示逻辑文件
- 日志展示组件
- 状态表格展示组件

## 10. 验收标准

- 输入框不再过大。
- 布局不再松散。
- 账号状态一眼可见。
- 成功 / 失败 / 运行中状态清楚。
- 不影响原有登录流程。
- 源码模式和 exe 模式都能打开界面。
- 没有修改稳定业务模块。

## 11. UI 改造执行流程

每次 UI 改造前，必须先输出：

1. 准备修改的文件列表
2. 本次 UI 改造目标
3. 明确不会修改的稳定模块
4. 是否涉及业务逻辑

每次 UI 改造后，必须输出：

1. 实际修改的文件列表
2. 改了哪些界面区域
3. 是否修改了业务逻辑
4. 是否影响 OCR、DM、Playwright、登录流程、打包流程
5. 源码模式验证方式
6. exe 模式是否需要重新打包验证
